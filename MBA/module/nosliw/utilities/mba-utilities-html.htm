ZmeY Information
This is a collection of fifteen ActiveX objects that implement CRC-32 and Adler-32 checksum algorithms, seven cryptographic hash (also known as message digest) algorithms, and four algorithms for calculating Hash-based Message Authentication Codes (HMAC).

The seven message-digest algorithms are:
  • MD4 and MD5 developed by RSA Data Security, Inc.,
  • SHA-1 developed by the National Institute of Standards and Technology (NIST) and by the National Security Agency (NSA),
  • RIPEMD-128, RIPEMD-160, RIPEMD-256, and RIPEMD-320 developed by Hans Dobbertin, Antoon Bosselaers, and Bart Preneel.

    HMAC algorithms are:
  • HMAC-MD5, HMAC-SHA1, HMAC-RIPEMD128, and HMAC-RIPEMD160 based, respectively, on MD5, SHA-1, RIPEMD-128, and RIPEMD-160.
  • Unified interface to the most popular checksum, cryptographic hash, and HMAC algorithms.
  • Includes implementation of RIPEMD family of algorithms, which are not available from the standard CryptoAPI providers.
  • Optimized in assembly language for maximum performance on sixth generation x86 processors.
  • Accessible from a wide range of programming languages and development environments.
  • Designed for optimal performance in both single-threaded and multi-threaded environment.
    • Hash algorithms take an input data stream of arbitrary length and produce a hash value of a fixed size (typically 128 or 160 bits). The hash value can be used:
  • To verify that the contents of files or messages has not been modified.
  • To store passwords. The cryptographic hash is "one-way", which means there is no other way to determine which password was used to generate the hash value or to find another password that produces the same hash value than attempting a "brute-force" attack.
  • As a part of more complex cryptographic schemes.

    CRC-32 and Adler-32 are not cryptographic hash algorithms, but they perform a similar operation: map a data of arbitrary length to a value of fixed size (32 bits). The main difference is that checksums are designed to be fast and cryptographic hash functions are designed to be secure. CRC-32 checksums are used in some popular archive formats such as RAR and ZIP to verify the integrity of the archived files.

    RIPEMD-256 and RIPEMD-320 are constructed from, respectively, RIPEMD-128 and RIPEMD-160. The security level of RIPEMD-256 is only guaranteed to be the same as that of RIPEMD-128, and similarly for RIPEMD-320 with respect to RIPEMD-160. Use the extended 256-bit and 320-bit functions when you need a longer hash value (e.g. as input to some cryptographic function) without needing a higher security level. To learn more about RIPEMD family of hash functions, visit COSIC\'s web site.

    Think of HMAC algorithms as of "personalized" (by supplying a secret key) cryptographic hash algorithms. Only the party that knows the key can create or verify message digests. It is recommended that the length of the key is at least equal to the length of the output of the algorithm (128 bits for HMAC-MD5 and HMAC-RIPEMD128, 160 bits for HMAC-SHA1 and HMAC-RIPEMD160). Using the same key for both encrypting the data and calculating the HMAC digest should be avoided. For more information see RFC 2104 and related documents.
